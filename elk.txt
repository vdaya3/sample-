  <h2 id="background">1. Background &amp; Stack Overview</h2>
  <p>The ELK Stack — comprising <strong>Elasticsearch</strong> (search &amp; analytics engine), <strong>Logstash</strong> (data ingestion), and <strong>Kibana</strong> (visualization) — is the industry-standard platform for log management, observability, and security analytics. Beats agents and the Elastic Agent further extend data collection capabilities.</p>
  <p>Elastic offers two distinct licensing tiers:</p>
  <ul>
    <li><strong>Open-source (free):</strong> Apache 2.0 core features, available on self-hosted deployments.</li>
    <li><strong>Elastic subscription (licensed):</strong> Adds enterprise security, machine learning, alerting, fleet management, and support — available through Elastic Cloud or self-managed with a subscription key.</li>
  </ul>

  <!-- Section 2 -->

  <h2 id="licensing">2. Licensing Landscape</h2>
  <p>In January 2021, Elastic moved Elasticsearch and Kibana from the Apache 2.0 license to a <strong>dual-license model</strong>: the Elastic License 2.0 (ELv2) and the Server Side Public License (SSPL). This change was significant:</p>
  <ul>
    <li>SSPL requires any company offering ELK as a managed/cloud service to open-source their entire stack — a condition that effectively bars use as a hosted offering.</li>
    <li>AWS and Google forked the project as <strong>OpenSearch</strong> in response, but OpenSearch lags in ML, alerting maturity, and UX compared to the Elastic-licensed stack.</li>
    <li>The open-source (free) tier of Elastic itself is <strong>feature-limited</strong> and does not include security, ML, or advanced monitoring by default.</li>
  </ul>

  <div class="callout callout-warning">
    <div class="callout-title">⚠️ Important</div>
    The "open-source" or "free" tier of ELK should not be confused with "full-featured". Core security features like TLS encryption, role-based access control, and SAML/SSO authentication are <strong>not available</strong> in the free tier as of Elasticsearch 8.x.
  </div>

  <!-- Section 3 -->

  <h2 id="gaps">3. Critical Gaps in Open-Source ELK</h2>
  <p>The following capabilities are either absent or severely limited in the free/open-source distribution:</p>

  <div class="feature-grid">

      <h4>Authentication (SSO/SAML)</h4>
      <p>LDAP, Active Directory, SAML, and OIDC/OAuth2 integrations require a subscription. Teams must manage local users only.</p>
    </div>
    <div class="feature-card">
      <div class="feature-card-icon">🤖</div>

    <div class="feature-card">

      <h4>Alerting &amp; Actions</h4>

  <h2 id="features">4. Key Features Unlocked by the Licensed Version</h2>

  <h3>4.1 Security &amp; Compliance</h3>
  <ul>
    <li><strong>TLS/SSL encryption</strong> at rest and in transit — mandatory for PCI-DSS, HIPAA, SOC 2, ISO 27001 compliance.</li>
    <li><strong>Role-Based Access Control (RBAC)</strong> — granular permissions per index, field, or document.</li>
    <li><strong>Audit logging</strong> — tracks who accessed or changed what, essential for compliance audits.</li>
    <li><strong>Single Sign-On (SSO)</strong> via SAML 2.0, OIDC, LDAP, and Active Directory — integrates with our existing identity provider (e.g., Okta, Azure AD).</li>
    <li><strong>IP filtering</strong> and <strong>API key management</strong> for service-to-service authentication.</li>
  </ul>

  <h3>4.2 Observability</h3>
  <ul>
    <li><strong>APM (Application Performance Monitoring)</strong> — distributed tracing, error tracking, and service maps with full correlation across logs, metrics, and traces.</li>
    <li><strong>Uptime &amp; Synthetic Monitoring</strong> — proactive availability checks and SLA reporting.</li>
    <li><strong>Infrastructure monitoring dashboards</strong> — out-of-the-box dashboards for Kubernetes, cloud services, and hosts.</li>
  </ul>

  <h3>4.3 Machine Learning &amp; Analytics</h3>
  <ul>
    <li><strong>Anomaly detection jobs</strong> — automatically surface unusual spikes, dips, or behavioral changes in logs and metrics.</li>
    <li><strong>Forecast capacity</strong> — predict storage and compute needs based on historical patterns.</li>
    <li><strong>NLP &amp; semantic search</strong> — power intelligent log analysis and search with transformer models.</li>
  </ul>

  <h3>4.4 Alerting &amp; Incident Management</h3>
  <ul>
    <li>Native <strong>Kibana alerting rules</strong> with support for threshold, anomaly, and custom query-based triggers.</li>
    <li><strong>Connectors</strong> to PagerDuty, Slack, Microsoft Teams, Jira, and email.</li>
    <li><strong>Case management</strong> built into Kibana — track incidents from detection to resolution.</li>
  </ul>

  <h3>4.5 Enterprise Reliability</h3>
  <ul>
    <li><strong>Cross-cluster replication (CCR)</strong> — replicate indices across data centers for HA and DR.</li>
    <li><strong>Snapshot Lifecycle Management (SLM)</strong> — automated, policy-driven backups.</li>
    <li><strong>Searchable snapshots</strong> — query cold data stored in object storage (S3/GCS/Azure) without full rehydration.</li>
    <li><strong>Dedicated support SLA</strong> — 24×7 Elastic support with guaranteed response times.</li>
  </ul>

  <h2 id="risk">7. Risk of Staying on Open Source</h2>

  <div class="callout callout-warning">
    <div class="callout-title">⚠️ Risk Assessment</div>
    Remaining on the open-source (free) tier of ELK Stack exposes the organization to the following risks:
  </div>

  <h3>7.1 Security Risks</h3>
  <ul>
    <li>All Elasticsearch indices are accessible without authentication by default — a significant attack surface if exposed (even internally).</li>
    <li>Inability to enforce encryption means log data (which may contain PII, credentials, or sensitive business logic) travels and rests in plaintext.</li>
    <li>No audit trail — impossible to prove who accessed or modified data during a security review or breach investigation.</li>
  </ul>

  <h3>7.2 Compliance Risks</h3>
  <ul>
    <li>Failing SOC 2 Type II, PCI-DSS, or HIPAA audits due to lack of encryption, access control, and audit logging.</li>
    <li>Regulatory fines and reputational damage in the event of a data breach attributed to a missing security control.</li>
  </ul>

  <h3>7.3 Operational Risks</h3>
  <ul>
    <li>No native alerting pipeline means incidents go undetected longer, increasing MTTR and business impact.</li>
    <li>No vendor support — community forums and GitHub issues become the only recourse during a production outage.</li>
    <li>No automated backups — a cluster failure or accidental deletion could result in permanent log data loss.</li>
  </ul>

  <h3>7.4 Scalability Risks</h3>
  <ul>
    <li>Without CCR or searchable snapshots, scaling to multi-region or cost-optimized tiered storage is not feasible.</li>
    <li>Manual agent management at scale (100+ hosts) becomes operationally untenable without Fleet.</li>
  </ul>

  <h2 id="recommendation">8. Recommendation &amp; Decision</h2>

  <div class="callout callout-success">
    <div class="callout-title">✅ Recommendation</div>
    Procure an <strong>Elastic Platinum or Enterprise subscription</strong> to meet our security, compliance, observability, and operational requirements. The licensed version is not optional — it is the minimum viable configuration for production use in our environment.
  </div>

  <h3>Suggested Tier: Elastic Platinum (or Enterprise for SIEM)</h3>
  <ul>
    <li>Covers all security, RBAC, SSO, ML, CCR, SLM, APM, and alerting requirements.</li>
    <li>Enterprise adds SIEM, unlimited ML nodes, and dedicated support escalation paths — recommended if security operations (SOC) is in scope.</li>
    <li>Elastic Cloud (managed) should be evaluated alongside self-managed to reduce operational overhead.</li>
  </ul>

    <h2>Summary</h2>
